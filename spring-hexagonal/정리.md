# Mockito Java Agent 설정 정리

## 배경: 통합 테스트 실패

`@SpringBootTest` 통합 테스트에서 아래 에러가 발생했다.

```
Caused by: java.lang.IllegalStateException: No root application context located
```

### 직접적인 원인

`SpringHexagonalApplicationTest`에서 `MockedStatic`을 닫지 않아 mock이 누수되었다.

```java
// 문제: MockedStatic이 닫히지 않음
@Test
void run() {
    MockedStatic<SpringApplication> mocked = Mockito.mockStatic(SpringApplication.class);
    SpringHexagonalApplication.main(new String[]{});
    mocked.verify(() -> SpringApplication.run(SpringHexagonalApplication.class, new String[]{}));
}

// 해결: try-with-resources로 자동 close
@Test
void run() {
    try (MockedStatic<SpringApplication> mocked = Mockito.mockStatic(SpringApplication.class)) {
        SpringHexagonalApplication.main(new String[]{});
        mocked.verify(() -> SpringApplication.run(SpringHexagonalApplication.class, new String[]{}));
    }
}
```

이 테스트가 먼저 실행되면 `SpringApplication` 클래스가 mock 상태로 남는다.
이후 `@SpringBootTest` 테스트들이 `SpringApplication.run()`을 호출할 때 mock이 가로채서
아무것도 반환하지 않아 컨텍스트 로드에 실패한다.

## Mockito Self-Attach 문제

### Mockito의 바이트코드 조작 방식

Mockito는 mock을 만들 때 **바이트코드 조작(instrumentation)**을 사용한다.
특히 `mockStatic()`처럼 이미 JVM에 로드된 클래스를 mock할 때는
`java.lang.instrument` API를 통해 실시간으로 클래스 바이트코드를 수정한다.

### 문제: Self-attaching 제한

Mockito는 원래 실행 중인 JVM 프로세스에 **동적으로 자기 자신을 attach**해서
instrumentation API를 획득했다(self-attaching).

Java 버전별 제한:
- Java 9+: self-attach 시 경고 출력
- Java 16+: JEP 396으로 기본 비활성화
- Java 21: 사실상 차단

```
OpenJDK 64-Bit Server VM warning: Sharing is only supported for boot loader classes
because bootstrap classpath has been appended
```

### 해결: `-javaagent`로 사전 등록

Self-attach 대신, JVM 시작 시점에 Mockito를 **공식 Java agent로 미리 등록**한다.

```
JVM 시작
  → -javaagent:mockito-core.jar 로 에이전트 미리 등록
    → Instrumentation API 획득 (self-attach 불필요)
      → 테스트 실행 (mockStatic 등 자유롭게 사용)
```

| 방식 | 시점 | Java 21 |
|---|---|---|
| Self-attach | 런타임에 동적으로 자기 자신에 attach | 차단됨 |
| `-javaagent` | JVM 부팅 시 에이전트 선등록 | 정상 동작 |

## Gradle 설정

```kotlin
// 1. mockito-core JAR 경로만 얻기 위한 전용 configuration
val mockitoAgent: Configuration = configurations.create("mockitoAgent")

dependencies {
    // 2. isTransitive = false: JAR 파일 자체만 가져옴 (경로 참조 목적)
    //    의존성은 이미 testImplementation(spring-boot-starter-webmvc-test)에 포함되어 있음
    mockitoAgent("org.mockito:mockito-core:5.21.0") { isTransitive = false }
}

tasks.withType<Test> {
    useJUnitPlatform()
    // 3. JVM 시작 시 mockito-core를 agent로 등록
    jvmArgs("-javaagent:${mockitoAgent.asPath}")
}
```

### 각 설정의 역할

**`configurations.create("mockitoAgent")`**
- Gradle 의존성 해석을 위한 전용 버킷(configuration) 생성
- `testImplementation` 등과 분리하여 JAR 경로만 독립적으로 참조하기 위함

**`isTransitive = false`**
- mockito-core의 transitive 의존성(byte-buddy, objenesis 등)을 가져오지 않음
- 이미 `spring-boot-starter-webmvc-test`에 Mockito 전체 의존성이 포함되어 있으므로 중복 방지
- agent 등록에는 JAR 파일 경로만 필요하기 때문

**`-javaagent:${mockitoAgent.asPath}`**
- `mockitoAgent.asPath`로 해당 configuration에 포함된 JAR의 절대 경로를 가져옴
- JVM이 시작될 때 mockito-core를 agent로 로드하여 instrumentation API 확보
